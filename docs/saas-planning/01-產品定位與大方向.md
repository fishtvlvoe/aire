> 參與者：Fish（老魚）+ Claude Opus

## 第 1 輪：三段產線架構

Fish 說明整體系統由三段串接：

1. **Input（開發中）**：另一個 SaaS 工具，從 Dcard、PTT、新聞、Google Trends 抓取時事熱門關鍵字，產出影片主題建議。

2. **產出製作（目前在做的）**：中間段，負責「拿到主題 → 生出內容」。三種產出：
   - 影片（VideoGo 負責）
   - 文字（另一個獨立系統）
   - 圖片（另一個獨立系統）

   影片類型分三種：
   - (a) 真人知識解說（Kling AI 數位人，目前在做）
   - (b) 產品或物件介紹（可用不同模型）
   - (c) MV 音樂娛樂類型（可用不同模型）

3. **多平台發布（第三階段）**：PostGo，串接各社群平台自動發布。之前有做過但還沒優化好。

整體定位：**AI 內容自動化產線**，不是單一軟體。

---

## 第 2 輪：自動化程度 — 半自動（B 方案）

討論了兩種模式：
- A）全自動：設好就不管，系統全部跑完
- B）半自動：每一步用戶確認

Fish 選擇 **B）半自動** — 用戶是導演，系統是劇組。

套用設計哲學分析：
- 血清素設計 → 偏 A（用完安心離開）
- 有主見的設計 → 偏 A（系統能決定的不問用戶）
- 課題分離 → 偏 B（內容品質是用戶的事）

最終決定：用戶掌控方向，系統執行苦工。

## 第 3 輪：7 個決策點的分工

列出一支影片從無到有的 7 個決策點，全部由用戶（導演）決定：

| 步驟 | 用戶做什麼 | 系統做什麼 |
|------|-----------|-----------|
| 1. 選主題 | 從 Input 推薦的熱門主題中選一個 | Input 系統推薦 5 個熱門主題 |
| 2. 選角色 | 選要用哪個角色（沈知序、未來新角色...） | 列出可用角色清單 |
| 3. 確認腳本 | 看一眼，OK 或改幾個字 | 自動生成腳本草稿 |
| 4. 確認分鏡 | 看一眼，OK 或調整 | 自動拆分鏡 |
| 5. 等待生成 | 等通知 | 呼叫 Kling API 生成影片 + 拼接後製 |
| 6. 看成品 | 看完決定要不要發 | — |
| 7. 選平台發布 | 勾選 YouTube / IG / TikTok 等 | PostGo 自動發布 |

白話總結：**用戶做 7 次「Yes / No / 改一下」的決定，中間的苦工全部系統做。**

## 第 4 輪：介面與目標用戶

### 介面策略

- **Web App（電腦版）= 主力介面**
  - 完整功能：選主題、選角色、看腳本、看分鏡、預覽影片（可看 4K/8K 細節）、確認發布、管理角色包
  - 原因：影片品質確認（8K/4K/1080p）在手機上體驗不好，需要電腦螢幕

- **LINE Mini App（手機）= 輕量介面**
  - 只看不做（或簡單操作）：收通知、快速確認 Yes/No、看進度
  - 不做：編輯腳本、調分鏡、看畫質細節

### 目標用戶

不只房仲，擴大到五類：

1. 房仲
2. 自媒體創作者
3. 業務
4. 中小型企業主
5. 製造業與生產業

他們的共同需求：用自己的臉（或虛擬人）做口播、產品介紹、MV，但他們不會寫程式、不會剪片。

### 產品定位

> 「不會剪片的人，也能量產專業影片」

### 改名決定

videogo → **VideoGo**（跟 BuyGo、PostGo、PayGo 同系列，好唸好記，語音辨識準）。改名工程等 SaaS 討論完再一起做。
