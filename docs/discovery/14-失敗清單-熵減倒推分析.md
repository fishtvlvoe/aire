# 失敗清單 — 熵減倒推分析

> 日期：2026-03-19
> 方法論：熵減倒推（列出所有失敗可能 → 逐一排除 → 剩下的就是該做的）
> 來源：BGO 核心框架引擎 1（熵減）+ WOOP 模型 + 事前驗屍法

## 核心問題

> **「這個產品有哪些方式會死掉？」**

---

## 一、產品層面的失敗（做出來沒人用）

### F01：客戶用了覺得不如自己的經驗

| 項目 | 分析 |
|------|------|
| 失敗場景 | AI 估價 900 萬，客戶憑經驗知道是 850 萬，系統反而誤導 |
| 根因 | 實價登錄延遲 1-3 個月 + AI 缺少在地經驗值 |
| 已排除？ | ⚠️ 部分 — 已知延遲問題，但沒有解法 |
| 排除方案 | 系統定位為「輔助參考」不是「取代判斷」；顯示資料時間戳 + 信心區間；讓客戶能輸入自己的經驗值覆蓋 AI 估價 |

### F02：客戶用了兩三次就不用了

| 項目 | 分析 |
|------|------|
| 失敗場景 | 新鮮感過了，覺得操作麻煩，回去用舊方法 |
| 根因 | 操作步驟太多 or 產出的報告不夠有用 |
| 已排除？ | ❌ 未排除 — 還沒有 UX 驗證 |
| 排除方案 | POC 階段就要讓客戶用真實物件測，觀察他的反應；操作必須 < 3 分鐘 |

### F03：報告太像 AI 生成的罐頭內容

| 項目 | 分析 |
|------|------|
| 失敗場景 | 每份報告看起來差不多，客戶覺得「這我用 ChatGPT 也能做」 |
| 根因 | Prompt 不夠精準，沒有結合在地資料 |
| 已排除？ | ❌ 未排除 — 還沒有 prompt 設計 |
| 排除方案 | 報告必須包含具體的地址、實價登錄成交資料、數字比較，不能只有空泛描述 |

### F04：客戶的物件類型超出系統能力

| 項目 | 分析 |
|------|------|
| 失敗場景 | 客戶丟進來一筆農地、工業用地、法拍屋，系統完全不會分析 |
| 根因 | 不同物件類型的分析邏輯差很多 |
| 已排除？ | ❌ 未排除 — 還沒確認客戶主要經手什麼類型 |
| 排除方案 | MVP 先鎖定住宅（公寓/大樓/透天），其他類型標示「不支援」 |

## 二、資料層面的失敗（輸入是垃圾，輸出也是垃圾）

### F05：實價登錄資料不夠用

| 項目 | 分析 |
|------|------|
| 失敗場景 | 客戶操作的區域太冷門，附近 3 個月內只有 2 筆成交，樣本數不夠 |
| 根因 | 冷門區域交易量低 |
| 已排除？ | ❌ 未排除 |
| 排除方案 | 樣本數不足時系統要警告「資料不足，僅供參考」；可擴大搜尋範圍（同行政區 → 同鄉鎮） |

### F06：拍照辨識率太低

| 項目 | 分析 |
|------|------|
| 失敗場景 | 客戶拍的謄本模糊、反光、角度歪，AI 辨識錯誤百出 |
| 根因 | 手機拍照品質不一 |
| 已排除？ | ⚠️ 部分 — 已決定模糊就重拍 |
| 排除方案 | 加上「辨識結果確認」步驟，讓客戶檢查 AI 辨識的內容對不對再繼續 |

### F07：AI 產生幻覺（瞎編資料）

| 項目 | 分析 |
|------|------|
| 失敗場景 | AI 說「附近有捷運站」但實際沒有；AI 編了一筆不存在的成交紀錄 |
| 根因 | LLM 的固有問題，會用合理但不真實的資料填充 |
| 已排除？ | ❌ 未排除 — 這是最大的技術風險 |
| 排除方案 | 所有數字必須來自資料庫查詢，不讓 AI 自己「推測」數字；分析文字和數據要分開處理 |

### F08：591 資料取不到導致分析不完整

| 項目 | 分析 |
|------|------|
| 失敗場景 | 只有歷史成交價，沒有目前市場開價，分析只有一半 |
| 根因 | 591 無合法 API |
| 已排除？ | ⚠️ 部分 — 已設計「客戶自行提供」模式 |
| 排除方案 | 客戶截圖/貼文字提供同區開價；系統明確標示「以下分析基於你提供的資料」|

## 三、商業層面的失敗（賺不到錢）

### F09：客戶覺得 10 萬太貴不付

| 項目 | 分析 |
|------|------|
| 失敗場景 | 客戶聽到報價就退縮 |
| 根因 | 他不確定花這個錢值不值 |
| 已排除？ | ⚠️ 部分 — 已設計分段收費 |
| 排除方案 | 先收 5 萬（簽約金），交付後再收 5 萬；或先做免費 demo 讓他看到效果 |

### F10：做完客戶的版本後 SaaS 化失敗

| 項目 | 分析 |
|------|------|
| 失敗場景 | 想賣給其他房仲，但沒人買 |
| 根因 | 其他房仲不知道有這個工具 / 不信任 / 覺得用 ChatGPT 就夠了 |
| 已排除？ | ❌ 未排除 — 但這是 Phase 2 的問題，不影響 MVP |
| 排除方案 | MVP 先服務好一個客戶，有成功案例後再推廣 |

### F11：維運成本超過客戶願意付的

| 項目 | 分析 |
|------|------|
| 失敗場景 | AI API 費用 + 主機 + 地政查詢每月要 5000，客戶只願付 3000 |
| 根因 | 費用估算不準 |
| 已排除？ | ⚠️ 部分 — 已確認地政免費查詢方案 |
| 排除方案 | MVP 用最省方案（免費地政查詢 + 最便宜的 AI API + 最便宜的主機）|

### F12：你自己的時間成本算不過來

| 項目 | 分析 |
|------|------|
| 失敗場景 | 2 個月做完收 10 萬，但後續客戶一直要求修改、加功能、維護，你的時間被無限佔用 |
| 根因 | 沒有定義「交付範圍」和「什麼算額外收費」 |
| 已排除？ | ❌ 未排除 |
| 排除方案 | 合約要寫清楚：交付內容、修改次數限制、超出範圍的計費方式、維運費包含什麼 |

## 四、法律與合規層面的失敗

### F13：被告「非法估價」

| 項目 | 分析 |
|------|------|
| 失敗場景 | 屋主覺得被低估，拿報告去告「你沒有估價師資格憑什麼出估價報告」 |
| 根因 | 台灣《不動產估價師法》限制 |
| 已排除？ | ⚠️ 部分 — 已知要加免責聲明 |
| 排除方案 | 報告不能出現「估價」兩字，改叫「市場參考分析」；加免責聲明；不提供具有法律效力的文件 |

### F14：個資法問題

| 項目 | 分析 |
|------|------|
| 失敗場景 | 系統存了屋主的個人資料（姓名、地址、財務狀況），被駭或外洩 |
| 根因 | 個人資料保護法 |
| 已排除？ | ❌ 未排除 |
| 排除方案 | 最小化儲存個資；謄本辨識後不存原始圖片；加密儲存 |

### F15：客戶用系統的分析結果去欺騙屋主

| 項目 | 分析 |
|------|------|
| 失敗場景 | 客戶拿著「AI 說只值 800 萬」去壓屋主價格，但實際值 950 萬 |
| 根因 | 系統被當成壓價工具 |
| 已排除？ | ❌ 未排除 — 這是使用者行為，難以控制 |
| 排除方案 | 使用條款寫明「本系統分析結果僅供參考」；報告加浮水印標註非正式文件 |

## 五、技術層面的失敗

### F16：系統做到一半做不下去

| 項目 | 分析 |
|------|------|
| 失敗場景 | 開發到一半發現技術棧選錯、架構不對、要重來 |
| 根因 | 沒有先做 POC 驗證 |
| 已排除？ | ⚠️ 部分 — 已規劃 POC 先行 |
| 排除方案 | POC 2 週內驗證核心技術（OCR + 實價登錄 + AI 分析），確認可行再往下 |

### F17：系統上線後掛掉沒人修

| 項目 | 分析 |
|------|------|
| 失敗場景 | 凌晨 2 點系統壞了，客戶隔天早上要用 |
| 根因 | 一人維運，沒有監控 |
| 已排除？ | ❌ 未排除 |
| 排除方案 | 用 serverless 架構降低維運負擔；加基本的錯誤通知（LINE Notify） |

### F18：實價登錄改格式或改網址

| 項目 | 分析 |
|------|------|
| 失敗場景 | 政府改了 CSV 格式或下載網址，資料匯入壞掉 |
| 根因 | 依賴政府外部資料 |
| 已排除？ | ❌ 未排除 |
| 排除方案 | 匯入程式加容錯機制 + 格式變更警報 |

## 六、人的層面的失敗

### F19：客戶期待值管理失敗

| 項目 | 分析 |
|------|------|
| 失敗場景 | 客戶以為系統能「自動幫他賣房子」，交付後覺得「就這樣？」 |
| 根因 | 沒有明確溝通系統能做什麼、不能做什麼 |
| 已排除？ | ❌ 未排除 |
| 排除方案 | 簽約前明確列出「系統能做」和「系統不能做」的清單 |

### F20：你和客戶的溝通成本太高

| 項目 | 分析 |
|------|------|
| 失敗場景 | 客戶不會描述需求，你花大量時間翻譯，開發進度延遲 |
| 根因 | 客戶不是技術人員 |
| 已排除？ | ⚠️ 部分 — 已設計「帶電腦去現場」的流程 |
| 排除方案 | 用 prototype（原型畫面）讓客戶指著螢幕說「要」「不要」，不用他描述 |

### F21：你自己的能力瓶頸

| 項目 | 分析 |
|------|------|
| 失敗場景 | 遇到技術問題卡住，Claude 也解不了 |
| 根因 | 一人團隊的固有限制 |
| 已排除？ | ⚠️ 部分 |
| 排除方案 | 選成熟的技術棧（不要用太新太冷門的）；遇到卡關先換方案不要硬幹 |

## 七、黑天鵝（無法控制但要知道）

### F22：房市崩盤，客戶自己都快倒了

| 項目 | 分析 |
|------|------|
| 可能性 | 低但非零 |
| 後果 | 客戶付不出尾款，也不需要系統了 |
| 能做的 | 分段收費降低風險 |

### F23：政府修法禁止 AI 用於不動產分析

| 項目 | 分析 |
|------|------|
| 可能性 | 極低 |
| 後果 | 產品直接不能用 |
| 能做的 | 關注法規動態 |

### F24：大型房仲突然推出同類產品且免費

| 項目 | 分析 |
|------|------|
| 可能性 | 中 — 但方向不同（他們面向消費者） |
| 後果 | 如果他們也做房仲端工具，我們就沒優勢了 |
| 能做的 | 跑快一點，先建立客戶黏性 |
