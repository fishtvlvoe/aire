## F03：報告太像 AI 罐頭內容

### 問題描述

每份報告看起來差不多，客戶覺得「這我用 ChatGPT 也能做」。

### 根本原因

1. Prompt 太通用，沒有針對個案輸入具體數據
2. AI 產出的文字缺少「只有這個物件才有的」細節
3. 沒有結合實價登錄的真實數字

### 什麼情況會發生？

- 客戶輸入兩個不同區域的物件，報告的「優劣分析」段落幾乎一樣
- 只是把地址和數字換掉，結構和措辭完全相同
- 客戶看三份報告後覺得「都長一樣」

### 排除方案

| 層級 | 做法 |
|------|------|
| 數據層 | 報告中的每個數字都來自真實資料（實價登錄），不是 AI 猜的 |
| 比較層 | 加入「跟同區域其他物件的比較」，每份報告的比較對象不同 |
| 客製層 | 加入物件特有的分析（例如：「此物件屋齡 5 年，比同區域平均 15 年新，這是賣點」） |
| 格式層 | 用卡片式呈現 + 數據圖表，視覺上不像「一段文字」 |

### 排除後狀態：⚠️ 部分排除

需要在 POC 階段用真實物件測試，確認報告品質。

## F04：物件類型超出系統能力

### 問題描述

客戶丟進一筆農地、工業用地、法拍屋，系統完全不會分析。

### 根本原因

1. 不同物件類型的估價邏輯差很多
2. 實價登錄的資料結構按物件類型分類，混在一起比會出錯
3. AI 的 prompt 沒有針對不同類型做差異化處理

### 什麼情況會發生？

- 客戶輸入一筆農地，系統用住宅的邏輯去分析，估出離譜的價格
- 客戶輸入法拍屋，系統不知道要打折估價
- 客戶輸入商辦，系統用住宅坪數去比，單價差異巨大

### 排除方案

```
系統辨識物件類型
  → 住宅（公寓/大樓/透天）→ 正常分析
  → 商辦 → 顯示「商辦物件分析模式」（簡化版）
  → 農地/工業/法拍 → 顯示「此類型暫不支援分析」
```

**MVP 只支援住宅。** 在 PRD 的 Out-of-Scope 明確寫上。

### 排除後狀態：✅ 已排除

只要在系統中限定支援類型 + 不支援的類型顯示提示，就不會出錯。

## F05：冷門區域資料不足

### 問題描述

客戶操作的區域太冷門，附近 3 個月內只有 2 筆成交，樣本數不夠做分析。

### 根本原因

1. 實價登錄的資料量取決於該區域的交易活躍度
2. 鄉下地區一年可能只有幾筆成交
3. 樣本太少的統計分析沒有意義

### 什麼情況會發生？

- 客戶輸入偏遠鄉鎮的物件，系統搜尋 500 公尺內的成交紀錄 → 0 筆
- 擴大到 1 公里 → 2 筆，但一筆是透天一筆是公寓，完全不能比
- 系統硬擠出一個「均價」但完全不準

### 排除方案

| 資料筆數 | 系統行為 |
|----------|---------|
| ≥ 20 筆 | 正常分析，高信心度 |
| 10-19 筆 | 正常分析，標示「資料量有限，僅供參考」 |
| 5-9 筆 | 簡化分析，明確警告「樣本不足」 |
| < 5 筆 | **不給估價**，只列出原始成交紀錄，讓客戶自己判斷 |

同時提供「擴大搜尋範圍」選項（500m → 1km → 同鄉鎮 → 同行政區）。

### 排除後狀態：✅ 已排除

只要有分級機制 + 資料不足時降級處理，就不會給出錯誤的估價。

## F17：系統掛掉沒人修

### 問題描述

凌晨 2 點系統壞了，客戶隔天早上要用。

### 根本原因

1. 一人開發 + 維運，沒有備援人力
2. 沒有監控機制，壞了都不知道
3. 傳統主機需要手動管理

### 什麼情況會發生？

- 主機記憶體滿了，服務掛掉
- SSL 證書過期，網站打不開
- AI API 的 API key 到期或額度用完
- 實價登錄下載排程失敗，資料停止更新

### 排除方案

| 做法 | 效果 |
|------|------|
| 用 Serverless 架構（Vercel / Cloudflare Workers） | 不用管主機，自動擴容，幾乎不會掛 |
| 加基本監控（UptimeRobot 免費方案） | 網站掛了立刻 LINE 通知你 |
| API key 到期提醒 | 設定日曆提醒，到期前 7 天更新 |
| 實價登錄下載加容錯 | 下載失敗時用上一次的資料，不會整個壞掉 |

### 排除後狀態：⚠️ 部分排除

Serverless 大幅降低風險，但不能保證 100% 不出問題。加監控通知是最低要求。

## F18：實價登錄改格式

### 問題描述

政府改了 CSV 格式或下載網址，資料匯入壞掉。

### 根本原因

1. 依賴外部政府資料，格式變更你無法控制
2. 政府改格式通常不會事先通知
3. 一改就壞，而且你可能幾天後才發現

### 什麼情況會發生？

- 內政部調整 CSV 欄位順序或名稱，匯入程式讀到錯的欄位
- 下載網址改了，排程抓到 404 錯誤
- 新增欄位或刪除欄位，程式跑出例外

### 排除方案

| 做法 | 效果 |
|------|------|
| 匯入前檢查 CSV 標頭（欄位名稱） | 格式變了立刻警告，不會用錯誤資料覆蓋正確資料 |
| 匯入失敗時保留上一次資料 | 資料最多延遲一期（10 天），不會清空 |
| 下載失敗時 LINE 通知 | 你知道壞了才能修 |
| 程式碼寫得有彈性 | 用欄位名稱讀取（不是用欄位位置），容忍順序變動 |

### 排除後狀態：⚠️ 部分排除

不能保證政府不改格式，但能確保改了之後系統不會壞掉，只是暫停更新。

## F20：你和客戶的溝通成本太高

### 問題描述

客戶不會描述需求，你花大量時間翻譯，開發進度延遲。

### 根本原因

1. 客戶是房仲不是技術人員，他說的「功能」跟你理解的不一樣
2. 口頭描述容易有落差
3. 他可能不知道自己要什麼，直到看到實物

### 什麼情況會發生？

- 他說「我要一個分析報告」，你做出來他說「不是這種」
- 你問他想要什麼，他說「好用的就好」，這無法執行
- 來回溝通 5 次才確認一個功能的細節

### 排除方案

| 做法 | 效果 |
|------|------|
| **帶電腦去現場**，讓他指著螢幕說「要/不要」 | 比口頭描述精確十倍 |
| **先做 Prototype（畫面原型）** | 他看得到東西才能給具體回饋 |
| **錄音 + 文字整理** | 他講的每句話都記錄，避免記憶落差 |
| **每次溝通後產出書面確認** | 你整理成文件，他看過確認，雙方同意才動手 |

### 排除後狀態：⚠️ 部分排除

你已經在做這件事（帶電腦去、把對話整理給我），但需要在整個開發過程中持續執行。

## F21：你自己的能力瓶頸

### 問題描述

遇到技術問題卡住，Claude 也解不了。

### 根本原因

1. 一人團隊的固有限制
2. 某些技術問題需要特定領域經驗
3. 不知道自己不知道什麼（未知的未知）

### 什麼情況會發生？

- 實價登錄 CSV 匯入的編碼問題搞不定
- 部署到雲端時環境設定出錯
- 效能問題（資料量大時查詢太慢）

### 排除方案

| 做法 | 效果 |
|------|------|
| 選成熟的技術棧 | Next.js + Supabase 或 Python + Streamlit，社群資源多 |
| 遇到卡關先換方案 | 不要在一個方法上死磕超過 2 小時 |
| 善用 AI 工具的多樣性 | Claude 解不了的問 Kimi、GPT、社群 |
| 降低技術複雜度 | MVP 能用就好，不追求最佳架構 |

### 排除後狀態：⚠️ 部分排除

不可能完全排除，但選對技術棧可以把風險降到很低。
