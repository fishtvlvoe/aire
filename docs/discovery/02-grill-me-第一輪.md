## 待回答的三個根本問題

### Q1：用戶是誰？

系統的目標使用者決定整個架構方向：

| 選項 | 架構影響 |
|------|----------|
| 單一房仲公司內部使用 | 單租戶、客製化、內部部署 |
| SaaS 賣給多家房仲 | 多租戶、訂閱制、雲端 |
| 獨立房仲個人工具 | 輕量級、低成本、快速上手 |

### Q2：市場行情資料從哪來？

沒有資料，整個系統就是空殼。台灣不動產資料來源選項：

| 來源 | 可行性 | 備註 |
|------|--------|------|
| 實價登錄（內政部） | 有公開資料 | 需確認 API 或爬蟲 |
| 591 / 信義房屋 | 需確認合法性 | 可能有使用限制 |
| 客戶自有資料庫 | 最理想 | 需確認是否存在 |
| Google Maps API | 可取得周邊設施 | 不含房價 |

### Q3：MVP 優先順序？

9 個輸出太多，建議先確認真正痛點排序。推測的優先級：

| 優先級 | 輸出項目 | 理由 |
|--------|----------|------|
| P0 必要 | 行銷文案 | 每天都在用，立即有感 |
| P0 必要 | 市場預期價格 | 跟屋主談判的核心武器 |
| P0 必要 | 議價策略 | 跟買方談判必備 |
| P1 重要 | SWOT 分析 | 專業感，但不是每天用 |
| P1 重要 | 屋主經營指南 | 有用但不緊急 |
| P2 加值 | 影片劇本/分鏡/提示詞 | 不是所有房仲都做影片 |
| P3 未來 | 歌曲 + MV | 很酷但優先級最低 |

## 還沒問但之後要追問的

- 預算和時程限制？
- 客戶的技術能力？（會不會自己操作系統）
- 是否已有現成工具在用？（要取代什麼？）
- 商業模式？（客戶付費 or 內部工具）
