# 對焦紀錄 — 第二輪

> 日期：2026-03-19
> 狀態：已確認

## 關鍵新資訊

### 你（顧問）的角色

- 前房仲出身，離開產業近 20 年
- 現為客戶的顧問，負責產品架構設計
- 客戶找你的原因：你懂流程 + 懂怎麼建構產品
- **不是外包開發**，是信任關係 — 客戶不想讓員工或外人接觸這些商業機密

### 系統定位修正

| 原本理解 | 修正後 |
|----------|--------|
| 報告給屋主看 | ❌ 報告是給**房仲自己看**的操作手冊 |
| 公開的 SaaS 工具 | ❌ **私人工具**，客戶不想讓員工或同業拿到 |
| 取代手動文書 | ✅ 但更核心的是**輔助判斷和談判** |

### 核心使用場景（修正版）

```
遇到賣家
  → 拍照（謄本/地址等）餵給系統
  → AI 辨識 + 自動帶入區域資料
  → 產出分析報告（房仲自己看）
  → 系統告訴房仲：
     ・這物件有沒有問題？（抵押、假扣押）
     ・開價合不合理？（比行情高多少%）
     ・如果堅持這個價，預估要賣多久？
     ・建議的出價區間
  → 房仲根據報告跟屋主談判
  → 簽委託 or 放棄
```

### 新發現的需求：物件風險檢查

客戶特別提到想知道：

| 檢查項目 | 說明 |
|----------|------|
| 貸款狀況 | 市價 1000 萬但又貸了 1000 萬出來 |
| 假扣押 | 物件是否被法院假扣押 |
| 抵押權 | 是否有設定抵押 |
| 其他限制 | 是否有其他產權問題 |

> 這些資訊理論上在謄本上可以看到，但系統能不能自動辨識並警示？待研究。

### 新發現的需求：即時對話分析

客戶希望：
- 與屋主對話時開啟錄音
- 系統即時分析屋主說的話是真是假
- 讓房仲能當場做判斷

> 這是進階功能，技術上涉及語音轉文字 + LLM 即時分析。MVP 可能先不做，但要納入規劃。

## 處理模式

| 項目 | 確認內容 |
|------|----------|
| 頻率 | 一次一個物件，遇到賣家才分析 |
| 數量 | 客戶手上約 10-20 件 |
| 報告呈現 | 房仲自己看，不給屋主 |
| 後續動作 | 條件好的物件 → 發布 591 做「黑箱測試」→ 觀察詢問度 → 帶看 |

## 預算與時程

| 項目 | 內容 |
|------|------|
| 預算 | 約 2-3 萬/月 |
| 時程 | 3 個月完成 |
| MVP 標準 | 能用就好，介面醜沒關係 |

## 技術選型（待定）

- 全客製 vs 低代碼 vs AI wrapper → 需了解客戶使用情境後決定
- 目前傾向：先做能跑的 POC

## 客戶作業流程

```
1. 取得物件資訊（謄本、地址等）
2. 判斷物件條件好不好
3. 若條件好 → 發布到 591 等平臺（黑箱測試）
4. 觀察是否有買方來電詢問
5. 安排員工或自己帶看
6. 成交
```

## 待問客戶的問題清單

> 以下問題需要直接問客戶才能確認

| # | 問題 | 為什麼要問 |
|---|------|-----------|
| 1 | 你拿到物件時，手上通常有什麼文件？謄本？名片？DM？截圖？ | 決定 AI 辨識要處理什麼格式 |
| 2 | 一天大概處理幾個新物件？ | 決定系統容量和收費模式 |
| 3 | 你現在判斷開價合不合理，是靠經驗還是有查資料？查什麼？ | 了解現有工作流，才知道要自動化什麼 |
| 4 | 你現在有在用什麼 AI 工具嗎？（ChatGPT、其他？） | 了解客戶的 AI 熟悉度 |
| 5 | 你手機是 iPhone 還是 Android？ | 影響 App 開發策略 |
| 6 | 報告你希望怎麼看？手機直接看？還是要能轉 PDF？ | 決定輸出格式 |
| 7 | 你底下有幾個員工？未來他們也要用嗎？ | 影響多帳號 / 權限設計 |
| 8 | 你操作的區域主要在哪？（臺南？跨縣市？） | 影響資料來源的涵蓋範圍 |
| 9 | 謄本你通常是紙本還是電子檔？ | 影響 OCR 辨識方案 |
| 10 | 「即時對話分析」這個功能，你想像中是怎麼用的？手機放桌上錄音？還是？ | 釐清即時分析的實際操作場景 |
