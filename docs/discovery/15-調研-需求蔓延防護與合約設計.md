# 調研：需求蔓延防護與合約設計

> 日期：2026-03-19
> 狀態：調研完成

## 需求蔓延的根本原因

### 客戶為什麼一直改需求？

| 心理機制 | 說明 |
|----------|------|
| 「看到才知道要什麼」 | 客戶看到初版後才發現「不是我要的」，這是人類認知限制 |
| 錨定偏差 | 「加這個應該很簡單吧」— 低估開發成本 |
| 沉沒成本心態 | 「都付錢了，多加一點沒差吧」 |
| 恐懼遺漏 | 擔心上線後缺功能，不斷追加「以防萬一」 |

### AI 開發的特殊陷阱

客戶知道你用 AI 寫程式很快 → 產生「既然這麼快，多做一點也不是問題」的心態。
**實際上 AI 加速的是編碼，不是理解需求、測試、部署的時間。**

### 前期調研不足 → 需求蔓延的因果鏈

```
前期調研不足
  → 需求文件模糊
  → 雙方對「完成」的定義不同
  → 開發到一半才發現遺漏
  → 客戶覺得「本來就該做」，開發者覺得「額外需求」
  → 衝突 or 默默吞下
  → 蔓延
```

**核心解法：把調研本身定義為付費交付物。**

## 收費結構建議（混合制）

| Phase | 計費方式 | 交付物 | 金額 |
|-------|---------|--------|------|
| Phase 1：需求調研 | 按時計費 or 固定 | PRD 文件 | 此階段結束後客戶可決定是否繼續 |
| Phase 2：開發執行 | 固定範圍 | 依 PRD 報價 | 超出 PRD 走變更流程 |
| Phase 3：維護 | 月費 or 按次 | bug 修復 + 小調整 | 上線後 30 天保固免費 |

## 付款里程碑

| 里程碑 | 交付物 | 金額比例 | 付款條件 |
|--------|--------|---------|---------|
| M0 簽約 | 合約簽署 | 30% | 簽約後 3 日內 |
| M1 設計完成 | PRD + UI 設計稿確認 | 20% | 驗收後 7 日內 |
| M2 開發完成 | 測試站功能展示 | 30% | 確認後 7 日內 |
| M3 上線驗收 | 正式站上線 | 20% | 驗收後 7 日內 |

**關鍵：每一里程碑之付款為下一階段開工之前提條件。未收款不超做。**

## 合約必備條款（台灣實務）

### 範圍鎖定

```
本契約之開發範圍以附件一「產品需求文件（PRD）」為準。
附件一「明確排除」所列項目不在本契約範圍內。
任何超出附件一範圍之需求，適用變更流程。
```

### 修改次數限制

```
本專案包含 3 次設計稿修改及 2 次功能調整。
超出次數之修改，依下列標準計費：
- 介面調整：每次 NT$ X,XXX
- 功能修改：每小時 NT$ X,XXX
所稱「一次修改」係指一次性提出之完整修改清單，非逐條分次提出。
```

### 驗收與默示通過

```
客戶應於收到交付物後 7 個工作天內完成驗收。
逾期未通知者，視為驗收通過。
驗收不合格者，應一次提出完整修改清單。
每階段驗收以 2 次為限，超過部分依變更流程另計費用。
```

### 付款遲延保護

```
逾期付款超過 15 日 → 開發方可暫停開發
逾期付款超過 30 日 → 開發方可終止契約，已收款項不退還
智慧財產權於付清全部款項後始移轉予客戶
付清前程式碼部署在開發方伺服器
```

## PRD 的「明確排除」怎麼寫

**比「功能清單」更重要的是「不做清單」。**

範例：
```
## 明確排除（Out-of-Scope）

以下功能不在本次開發範圍，如需追加請走變更流程：

- ❌ 手機原生 APP（本次僅 RWD 網頁）
- ❌ 即時對話錄音分析
- ❌ 自動爬蟲抓取 591 / 永慶資料
- ❌ 行銷影片 / 歌曲生成
- ❌ 多使用者帳號管理
- ❌ 既有資料遷移
- ❌ 正式的不動產估價報告
```

## 對本專案的具體建議

1. **PRD 就是你的防護盾** — 前期調研做得越紮實，後面被改的風險越低
2. **「不做清單」要寫進合約** — 客戶原始需求的 9 個輸出中，MVP 只做 4 個，另外 5 個要明確寫進 Out-of-Scope
3. **部署在你的伺服器** — 付清尾款才遷移，保護自己
4. **修改次數寫死** — 3 次設計修改 + 2 次功能調整，超過加錢
