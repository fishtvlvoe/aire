# 調研：AI 幻覺防護機制

> 日期：2026-03-19
> 狀態：調研完成

## 核心結論

> **LLM 是翻譯官，不是分析師。所有數字來自確定性計算，LLM 只負責把結構化數據翻譯成人話。**

## AI 幻覺的六種類型（在房地產場景）

| 幻覺類型 | 舉例 | 危害程度 |
|----------|------|---------|
| 數值捏造 | 「該社區均價 850 萬」但資料庫沒這筆資料 | 🔴 最高 |
| 數值偏移 | 實際 780 萬，AI 說 850 萬（混淆區域或時段） | 🔴 高 |
| 時間錯位 | 用 2023 資料回答 2026 行情 | 🟡 中高 |
| 計算錯誤 | 總價/坪數算出的單價有誤差 | 🟡 中 |
| 來源虛構 | 「根據內政部 2025 年第三季報告...」但報告不存在 | 🟡 中 |
| 統計方法錯用 | 平均數和中位數混用 | 🟡 中 |

研究數據：即使最先進的模型，在複雜金融推理中幻覺率仍有 **10-20%**。

## 解法：三層防護架構

```
┌─────────────────────────────────────┐
│  Layer 1: Input Guardrails          │  ← 問題是否在系統能力範圍內？
├─────────────────────────────────────┤
│  Layer 2: RAG + Grounding           │  ← 只用可信來源回答
├─────────────────────────────────────┤
│  Layer 3: Output Validation         │  ← 輸出的數字能追溯到來源嗎？
│  - 數值範圍檢查                      │
│  - 來源比對                          │
│  - 計算覆核                          │
│  - 時間一致性                        │
└─────────────────────────────────────┘
```

Stanford 研究：RAG + RLHF + Guardrails 三者結合，幻覺率可降低 **96%**。

## 關鍵設計原則

| 原則 | 做法 |
|------|------|
| 數字不靠 LLM 算 | 所有數值用 SQL/Python 確定性計算，LLM 只做自然語言包裝 |
| 每個數字有身份證 | 每個輸出數字綁定來源紀錄 ID + 計算方法 |
| 用戶可驗證 | 前端「展開來源」功能，顯示原始資料和計算過程 |
| 自動比對 | 驗證層自動檢查 LLM 輸出的數字是否與計算結果一致 |
| 分級信心度 | 資料充足（>20 筆）→ 高信心；稀少（<5 筆）→ 標示警告 |
| 時間標記 | 明確標示資料時間範圍 |
| 降級機制 | 驗證失敗不顯示數字，改為「資料不足，無法估算」 |

## 「一鍵驗證」的具體呈現

```
┌──────────────────────────────────────┐
│  「附近成交均價 [850萬]¹」            │
│                                      │
│  ¹ 點擊展開：                        │
│  ┌──────────────────────────────┐    │
│  │ 來源：實價登錄 2025Q4          │    │
│  │ 範圍：東區裕農路 500m 內       │    │
│  │ 筆數：23 筆                    │    │
│  │ 原始數據：                     │    │
│  │  - 裕農路 30 號 / 42坪 / 920萬│    │
│  │  - 裕農路 45 號 / 38坪 / 810萬│    │
│  │  - ...（共 23 筆）             │    │
│  │ 計算方式：算術平均              │    │
│  └──────────────────────────────┘    │
└──────────────────────────────────────┘
```

## 後端處理流程

```
1. 用戶提問 →
2. RAG 從實價登錄 DB 檢索相關紀錄 →
3. Python 確定性計算（均價、中位數、單價）→
4. LLM 把計算結果翻譯成自然語言 →
5. 驗證層比對 LLM 輸出的數字 = 計算結果 →
6. 通過 → 顯示；不通過 → 重新生成或降級
```

## 業界參考

| 公司 | 做法 |
|------|------|
| HouseCanary | 估價核心是統計模型不是 LLM，LLM 只做翻譯 |
| Zillow | 核心是 ML 迴歸模型，AI 用於搜尋和語言理解 |
| 金融業通用 | 確定性邏輯驗證引擎 + Human-in-the-loop |

## 對本專案的影響

### 設計結論

1. **數字計算不能交給 AI** — 用 Python/SQL 算，不讓 LLM 猜
2. **每個數字都要有來源** — 客戶點一下就能看到原始資料
3. **信心度要標出來** — 資料不足時主動告知，不硬給數字
4. **LLM 只做翻譯** — 把結構化數據翻成人話，不做推理

### 這解決了 F07（AI 幻覺）

排除方案確認：
- ✅ 數字來自確定性計算 → 不會瞎編
- ✅ 每個數字有來源 → 客戶可驗證
- ✅ 資料不足就降級 → 不會硬給不準的數字
- **F07 從「未排除」升級為「已排除」**
